What gstack Gets Right (And What It Misses When You're Actually Shipping Product)

Garry Tan's gstack validates the role-based agent pattern. But when you're shipping real product solo, the hard part is everything outside the framework's scope.

Garry Tan open-sourced gstack on March 12. Over 40,000 GitHub stars in under two weeks. TechCrunch coverage. The usual cycle: excitement, backlash, hot takes about whether it's just "a bunch of prompts."

I've been running a similar workflow since mid-2025—skills files, role-based agent behavior, specs-driven development—while shipping a real product to a paying client. Not as a side project. As the entire business.

gstack validates the pattern. It also reveals what gets lost when the conversation stays at the framework level and never touches production.

The Core Insight Is Correct

The idea behind gstack is sound: when you give an AI agent a single generic role, output quality degrades. When you assign it a specific persona with constrained responsibilities—a reviewer who only reviews, a planner who only plans—the output improves dramatically.

This maps to how real engineering organizations work. Different cognitive gears for different phases. A QA engineer thinks differently from a product lead. A release manager applies different judgment than a developer writing feature code. gstack codifies that separation into slash commands.

I arrived at the same conclusion independently, through a different path. Not by thinking about organizational structure, but by watching agents produce code that compiled, passed basic checks, and felt subtly wrong everywhere. The patterns drifted. The naming was inconsistent. The architecture choices were locally reasonable and globally incoherent.

Skills files were the fix. Detailed instruction sets—design patterns, Quarkus conventions, React component standards, error handling rules, module boundary definitions—loaded at the start of every session. The output went from "plausible code for a generic project" to "recognizably my codebase."

The underlying principle is the same as gstack's: constrain the agent's operating context, and it performs better. We just implemented it differently.

Where gstack Stops

gstack is a development process tool. It covers the lifecycle from ideation through deployment: think, plan, build, review, test, ship, reflect. Within that scope, it's well-designed. The /plan-ceo-review skill that asks "what's the 10-star product hiding inside this request" is a genuinely useful forcing function.

But when you're a solo founder shipping product, the engineering workflow is maybe 40% of what agents need to do for you. The other 60% is domain-specific work that no generic framework can anticipate.

Here's what I mean concretely.

Domain Context Is the Harder Encoding Problem

My skills files don't just encode engineering patterns. They encode business logic.

When I'm running event marketing campaigns for a client, agents need to understand ICP definitions—what makes a company a priority target versus a secondary one. They need to know how outreach sequences work across persona segments. They need to understand CRM field mappings, lead scoring logic, and attribution models.

None of this is engineering knowledge. It's domain knowledge. And it changes constantly—new events, new target lists, shifting priorities from the client.

gstack's /plan-ceo-review can ask good product questions. But it can't ask "does this outreach sequence match the methodology we agreed on" or "is this lead classification consistent with the ICP criteria we defined for this specific event." That judgment has to come from somewhere, and for me it comes from skills files and specs that encode business decisions, not just engineering conventions.

The encoding labor for domain context is harder than encoding engineering patterns. Engineering conventions are relatively stable—your error handling patterns don't change week to week. Domain context shifts with every new client conversation, every new event, every updated target list. Keeping those instructions current is a significant, ongoing cost that gstack's model doesn't account for.

Specs Are the Missing Coordination Layer

gstack assumes you know what to build. Its process starts at "think about the product" and moves through implementation. What it doesn't address is the living document problem—how do you maintain a single source of truth that agents can reference across sessions, across days, across weeks?

I use a specs repo. Not a task board. Not Jira. The specs repo contains detailed business requirements broken into workflows, specific tasks with implementation details, data model specifications, integration contracts, UI/UX requirements, and testing criteria.

After every PR merge, specs get updated. This is non-negotiable. The moment specs drift from the codebase, agents start making decisions based on outdated assumptions. You don't notice it immediately. Features still ship. Then something breaks in an unexpected place because the agent was building against a version of the system that no longer exists.

gstack's skills are stateless. Each slash command runs in its own context. That's fine for a single feature cycle. It's insufficient for a product that's been under continuous development for months, where decisions made in November constrain what's possible in March, and agents need to understand that history.

Parallel Agents Need Isolation, Not Roles

gstack's model is sequential. Think, plan, build, review, test, ship. One agent, switching roles.

My workflow runs multiple agents in parallel. Cursor building a frontend feature while Claude Code refactors a backend module. Overnight agents tasked across separate repos—one running test suites, another cleaning up integrations, a third handling a data migration—each producing a discrete PR by morning.

The key constraint isn't roles. It's isolation. If two agents touch overlapping parts of the codebase, you wake up to merge conflicts and divergent assumptions. The work has to be cleanly separated: different branches, different modules, different repos.

This is an orchestration problem, not a persona problem. gstack doesn't address it because gstack is designed for one person doing one thing at a time. When you need to compress a week of team output into two days of solo work, parallel execution is where the real leverage is.

The 10,000 Lines Per Week Metric Is Misleading

Garry Tan reported averaging 10,000 lines of code and 100 pull requests per week over a 50-day period. The README makes even bigger claims—10,000 to 20,000 lines per day, 600,000+ lines in 60 days. This generated both excitement and skepticism.

I won't question his numbers. I will say that lines of code is the wrong metric for product development.

When I'm building Luminik, the most valuable days are often the ones with the fewest lines committed. Days spent rewriting specs after a client conversation reveals a wrong assumption. Days spent deleting code because a simpler architecture emerged. Days spent manually testing every UI flow because agents consistently miss edge cases in multi-step interactions.

The work that matters most—deciding what to build, maintaining system coherence, encoding domain judgment, verifying quality—doesn't show up in LOC counters or PR counts. Optimizing for code output volume risks optimizing for the wrong thing.

What Actually Matters for Solo Builders

If you're building alone with AI agents, here's what I'd focus on instead of choosing between gstack and alternatives:

Encode your judgment, not just engineering patterns.

Skills files should capture how you think about your specific domain, not just how you want code structured. The business logic, the customer context, the decision frameworks—these are what make agent output actually useful versus generically correct.

Maintain a living coordination layer.

Call it specs, call it a design doc repo, call it whatever. The point is that agents need access to the current state of the entire system's intent, not just the code. Without it, every session starts from scratch.

Design for parallel execution.

The biggest leverage in solo development isn't making one agent faster—it's running multiple agents simultaneously on cleanly separated work. This requires thinking about your codebase in terms of independent modules with clear boundaries.

Measure outcomes, not output.

The question isn't how many lines you shipped. It's whether the product works, whether customers use it, and whether the system stays coherent as it grows.

gstack is a useful contribution. It's well-packaged, it's open source, and it validates a pattern that works. But frameworks optimize for the framework's scope. The actual challenge of building product alone is everything outside that scope—the domain knowledge, the coordination, the judgment calls that no slash command can make for you.

That's where the real work is. And it's the part nobody's figured out how to automate yet.
