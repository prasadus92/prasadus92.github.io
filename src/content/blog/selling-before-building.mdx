<p class="reveal">
  In Luminik's early months I did not lead with a product walkthrough. I sent a narrow outbound message to a marketing leader whose team cared deeply about events, then offered to do the work for them. The work became real before the software did.
</p>

<KeyTakeaway label="The argument in one paragraph">
  If you can sell a manual version of the outcome, you learn faster than any prototype can teach you. The repeated parts of the work become the product. The non-repeated parts stay with humans. Build that boundary first, then write the code.
</KeyTakeaway>

<h2 class="reveal">Why I Did It This Way</h2>

<p class="reveal">
  At Mainteny I took the conventional path: built the MVP first, raised a $2.7M seed, then kept learning what the market wanted. Funding does not answer the questions that make a company real. It does not tell you who will buy, which channel opens the door, what language the buyer uses, where urgency comes from, or which part of the workflow is expensive enough to fix. With SnowOptix I approached investors too early and felt the cost of the same mistake.
</p>

<h2 class="reveal">The First Useful Sentence Came From the Buyer</h2>

<p class="reveal">
  I had been thinking about automation. The buyer kept returning to the same idea: every event had its own quirks, and the value was not in automation by itself. The value was handling the messy middle well.
</p>

<PullQuote attribution="The first marketing leader I sold to, paraphrased">
  Different events, different attendee sources, different platforms, different reps, different follow-up rules. We do not need a one-click feature. We need someone to take responsibility for the messy middle.
</PullQuote>

<p class="reveal">
  That sentence reshaped the company. I stopped trying to sound like a software vendor and started listening for the work.
</p>

<h2 class="reveal">The Manual Loop That Became Luminik</h2>

<p class="reveal">
  The first events were real, time-bound, and operationally messy. I had no software surface yet. I had the customer's existing tools, a spreadsheet, and a lot of manual effort. From the buyer's point of view, I was removing work without asking them to change how they already operated.
</p>

<h2 class="reveal">What Being Embedded Taught Me</h2>

<p class="reveal">
  Several events overlapped and the work had to coordinate across sales stakeholders at the same time. It was too much for one person to keep doing manually, which was exactly the point.
</p>

</p>

<h2 class="reveal">From Pilot to Contract</h2>

<p class="reveal">
  By July, the work had stopped being speculative. The customer had seen me operate across real events, inside their real tools, with their real team. When the contract was signed, the value was already felt, because the product promise had been delivered manually first. The value was not a dashboard. It was an operating system for an event motion that had been depending on too much coordination.
</p>

<h2 class="reveal">When Software Finally Made Sense</h2>

<p class="reveal">
  I wrote the first code after enough delivery to know which parts deserved software. If I had built first, I would have designed from imagination. I would have overbuilt the clean parts and underbuilt the annoying parts. The annoying parts are usually where products live.
</p>

<p class="reveal">
  Luminik is still being built from that posture. Backend automation came first because it reduced delivery load immediately. The customer-facing UI follows the same rule: build the surfaces that make the workflow clearer, faster, and easier to trust.
</p>

<Callout tone="note" title="The operating rule I use now">
  <ol>
    <li>Find a buyer with a live operational problem.</li>
    <li>Listen for the words they use when the workflow breaks.</li>
    <li>Do the work manually inside their current process.</li>
    <li>Notice which steps repeat across events, reps, and tools.</li>
    <li>Charge once the manual work is clearly useful.</li>
    <li>Write software around the repeated work.</li>
  </ol>
</Callout>

<h2 class="reveal">What I Would Tell a Technical Founder</h2>

<p class="reveal">
  If you can sell a manual version of the outcome, you learn faster than a prototype can teach you. You also learn which promises you are willing to be accountable for. That accountability keeps the product honest.
</p>

<KeyTakeaway label="My bias now">
  Sell the outcome manually until the repeated work becomes impossible to ignore. Then write the software around the repeated work. The boundary you find this way is harder to fake and harder to lose.
</KeyTakeaway>
