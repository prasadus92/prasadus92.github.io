<div class="reveal">
  A solo founder loses time in small places. You reopen the same pull requests. You check the same issue list. You read the same logs twice because you forgot what changed overnight. You carry little pieces of the company in your head until they start bumping into each other.
</div>

<div class="reveal">
  Alfred exists to reduce that waste. It turns repeated work into the things I can actually use: a plan, a summary, a draft, a pull request summary, or a short list of decisions.
</div>

<div class="reveal">
  The screenshots in this post are real examples from my Alfred Slack channel. I am using them because they show the shape of work I want from agents: concrete, bounded, sourced, and waiting for a human decision.
</div>

<KeyTakeaway label="The model in one paragraph">
  Alfred is useful when it acts like a good chief of staff for repeated work: short, specific, sourced, and aware of what it is allowed to do. It can summarize, draft, plan, open work for review, and ask for approval. It does not publish, send, pay, merge, or deploy.
</KeyTakeaway>

<h2 class="reveal">What Alfred is</h2>

<div class="reveal">
  Alfred is my public name for a group of named agents around Luminik and my own work. Some act like company functions: chief of staff, content, sales, finance ops, and product ops. Some sit inside engineering: planner, implementer, reviewer, health checker, and notifier.
</div>

<div class="reveal">
  The names can be playful. The job descriptions have to be boring. Every role has to answer five questions before I trust it with repeated work.
</div>

<ol class="reveal">
  <li><strong>What can it read?</strong> Specific repos, files, public docs, approved notes, or logs required for the job.</li>
  <li><strong>What can it do?</strong> Search, summarize, draft, open a pull request, comment, or post a message for me to read.</li>
  <li><strong>Where does the work show up?</strong> A pull request, a review note, a markdown file, a Slack message, or a digest.</li>
  <li><strong>What evidence does it provide?</strong> Source links, pull request numbers, issue numbers, log lines, file paths, or screenshots approved for public use.</li>
  <li><strong>What requires my approval?</strong> Anything external, irreversible, expensive, customer-facing, or reputation-bearing.</li>
</ol>

<div class="reveal">
  If an answer is fuzzy, the role gets smaller. If the role cannot get smaller and still be useful, I delete it.
</div>

<h2 class="reveal">A job description is different from a prompt</h2>

<div class="reveal">
  A prompt asks a model to do something right now. A job description tells the agent what it owns, what it may read, and where the work should go. That distinction sounds minor until the work repeats.
</div>

<h2 class="reveal">What I want back</h2>

<div class="reveal">
  I do not want long agent transcripts. I want something I can read quickly. If the plan is good, I can approve it. If the summary is weak, the gap should be obvious.
</div>

<Callout tone="info" title="My rule">
  Alfred can prepare work. I still decide what gets sent, merged, published, paid, or deployed. I know it is working when I spend less time gathering context and more time making the actual decision.
</Callout>

<h2 class="reveal">Why I run it on real work</h2>

<div class="reveal">
  Agent demos often look impressive because the task is isolated. Company work is messier. The answer depends on old decisions, repo conventions, customer language, investor-safe wording, product limits, and what changed yesterday.
</div>

<div class="reveal">
  A single chat window cannot hold that for long. Alfred makes repeated work easier to run: one role for the job, one source list, one format for the answer, and one approval path.
</div>

<div class="reveal">
  This matters more when you are building alone. In a team, someone else may remember a customer call, question a pull request, or ask why a task matters. When I am the only operator, those reminders have to be built into the system.
</div>

<h2 class="reveal">How the engineering side works</h2>

<div class="reveal">
  Engineering is where Alfred became most concrete first. Luminik has separate repos for backend, frontend, mobile, data acquisition, specs, integrations, and agents. A feature often touches more than one of them. The failure mode is obvious: one repo moves, the others drift.
</div>

<div class="reveal">
  That is why I split the engineering work into named roles. Batman coordinates multi-repo work. Lucius and Robin can take bounded implementation tasks. Ra's al Ghul and Bane are useful as reviewers and pressure-testers. Nightwing watches the status and turns changed work into summaries.
</div>

<div class="reveal">
  The names are less important than the separation. The coordinator should not quietly become the implementer. The implementer should not be the final reviewer. The reviewer should not merge. The status role should not turn a weak signal into a decision.
</div>

<h2 class="reveal">Slack is where Alfred reports</h2>

<div class="reveal">
  I use Slack because it forces compression. The message has to be readable. It has a timestamp. It can link out to GitHub, logs, docs, or a draft. It can sit next to my approval instead of being buried inside a long agent transcript.
</div>

<div class="reveal">
  It also gives me a record. A normal morning summary tells me which agents ran, which pull requests opened, which pull requests merged, and which issues need attention. I can still go to GitHub for the full truth. Slack tells me where to look first.
</div>

<div class="reveal">
  Sales is important here. I like agents for preparation: summarizing context, finding the last conversation, reminding me what a buyer cared about, and drafting a next step. I do not want them to pretend they can own the relationship. A founder-led sale depends on hearing the buyer's world clearly. If an agent removes that contact, it is hurting the company.
</div>

<div class="reveal">
  This is also where <a href="https://riffon.com/insight/ins_4x3awd818syf">Rob Snyder's demand-side thinking</a> fits my own experience. The job is to find demand that already exists in the customer's world: a project, a blocked priority, a bad workaround, a to-do item someone is already trying to move. Alfred can help me prepare for those conversations. It cannot manufacture demand, and I do not want it trying.
</div>

<h2 class="reveal">Where Alfred goes wrong</h2>

<div class="reveal">
  Alfred gets worse when the role is allowed to be vague. "Help with engineering" is too wide. "Review the changed backend files in this pull request for data-contract risk and return comments only" is narrow enough to be useful.
</div>

<div class="reveal">
  It also gets worse when old context stays alive. A stale spec is dangerous because the agent follows it with confidence. The fix is boring and necessary: update the source docs, make agents cite them, and treat unsupported claims as defects.
</div>

<div class="reveal">
  The third mistake is too much automation. Some tasks are expensive because they require taste, trust, or a relationship. Customer replies are the obvious example. If I need to read the whole thread before approving the draft, I may as well write the reply myself.
</div>

<h2 class="reveal">The OSS version starts small</h2>

<div class="reveal">
  I extracted the reusable runtime as <a href="https://github.com/luminik-io/alfred-os" target="_blank" rel="noopener">Alfred OS</a>. The private version has company context, customer data, and adapters that stay private. The public version is useful without any of that.
</div>

<ul class="reveal">
  <li><strong>Job-description templates.</strong> Markdown templates for chief of staff, content, sales, ops, and engineering roles.</li>
  <li><strong>Repo instruction templates.</strong> Example AGENTS.md, CLAUDE.md, and content-guide files for a solo product team.</li>
  <li><strong>Review rules.</strong> Approval rules for code, content, sales replies, finance actions, and deployments.</li>
  <li><strong>Summary formats.</strong> Plans, digests, pull request summaries, risk summaries, and morning briefs.</li>
  <li><strong>A small local runner.</strong> Enough structure for scheduled jobs and work I can review without my private adapters.</li>
</ul>

<div class="reveal">
  A founder should be able to borrow the job descriptions even if they use Claude Code, Cursor, gstack, Codex, or a homegrown harness. The role matters more than the current tool.
</div>

<h2 class="reveal">Start with one repeated job</h2>

<div class="reveal">
  If you are building alone, start with one job you already repeat every week. Write the five questions. Keep the answer short and sourced. Keep approval with you.
</div>

<div class="reveal">
  Run it for a week. If it saves attention, automate the trigger. If it creates another thing to review, delete it. The goal is simple: spend less of your day rebuilding context you already paid to learn.
</div>

<KeyTakeaway label="The standard I use now">
  Alfred is good when repeated work becomes visible, sourced, and easy to reject. It is bad when the system hides uncertainty or asks for trust it has not earned. I keep working on that line.
</KeyTakeaway>
